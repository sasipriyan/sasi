Research Paper

IJERT · Vol. 15 Issue 01 · January 2026
Energy-Efficient AI Accelerator Architecture for Edge Devices
Start Here — The Hook
Think about a drone flying autonomously, a smart security camera detecting faces in real time, or a wearable fitness device monitoring your health. All of these devices need to run AI — but they run on tiny batteries, have very limited memory, and cannot afford to heat up. The question is: how do you run powerful AI on a device that has almost no resources? That is the exact problem our paper addresses.
Root Cause Discovery
When we looked at traditional AI chip designs, we found that more than 60% of total energy is not spent on actual computation — it is spent on moving data back and forth between the processor and main memory. So the problem is not the computation itself. The problem is the data movement. Once we identified that, we designed our architecture around solving exactly that.
Three Core Ideas We Proposed
Idea 1 — Data Locality
Keep data close to the processor
We designed a three-tier memory system. L1 — fast local buffer right at each processing unit, single cycle access. L2 — shared scratchpad for a cluster. L3 — global on-chip SRAM for weights and activations. Data rarely needs to travel far. This cuts energy significantly.
Idea 2 — Sparsity Exploitation
Skip zero-value computations entirely
In real neural networks, a large number of weights and activations are zero. Multiplying by zero is always zero — so those computations are pointless. We built a Sparsity Engine that detects zeros before computation and skips those operations entirely. No wasted compute. No wasted energy.
Idea 3 — Adaptive Dataflow
Adapt execution strategy per layer
Different neural network layers have different access patterns. Traditional accelerators use one fixed strategy — which is inefficient. Our architecture adapts per layer — selecting weight stationary, output stationary, or row stationary — so every layer runs as efficiently as possible.
Validation
We tested against MobileNetV2 and ResNet-18. Compared against conventional fixed-dataflow accelerators and NVDLA-style industry-grade edge accelerators. Results showed lower power consumption and faster inference compared to both baselines.
My Contribution
I contributed to the architectural design analysis — specifically how the three-tier memory hierarchy and dataflow controller interact. I worked on performance comparison studies and drafted the sections on optimization strategies and evaluation metrics.
Why is this relevant to Bosch?
BGST works on automotive ECUs, IoT sensors, embedded AI systems, ADAS — all resource-constrained edge devices. Every smart sensor in a Bosch vehicle faces exactly the problem our paper solves. Running AI efficiently on hardware with limited power and memory.
Why IJERT specifically?
IJERT is a peer-reviewed, indexed journal that aligned with our research domain and our timeline for a first-year M.Tech publication. The peer review process validated the quality of our work externally.
Was this fully implemented in hardware?
The current work is based on cycle-accurate simulation using CACTI and synthesized gate-level models on a 28nm CMOS process. The next step in the roadmap is FPGA or ASIC prototype implementation — which is our future work section.
💡 Closing line: "We rethought how AI chips manage data — keep it close, skip useless computations, adapt per layer. It is a foundational problem for the next generation of intelligent embedded products."

Samsung PRISM Research

Samsung Research Institute Bangalore · PRISM Program · Project 26LAI01VITC
Machine Unlearning in Keyword Spotting Systems
Start with the hook
Imagine you trained an AI model that can recognize your voice. Now you say — "I want my voice removed from your system." Under GDPR, you legally have that right. The obvious answer is to retrain the entire model from scratch. But with millions of users, that takes days of compute and is completely impractical. So the real question is: can we make an AI model surgically forget one specific person — without retraining everything, and without affecting anyone else? That is machine unlearning. And this is a Samsung-funded research project.
The Setup
We used the VoxCeleb dataset — 40 speakers, 4,874 voice clips. We used Wav2Vec2 — Facebook's state-of-the-art speech transformer model. After training, the model identified our target speaker with 99.37% confidence. That is the starting point. Now we need to make it forget.
Stage-by-Stage Breakdown
Stage 1
Baseline Model Training
We trained the Wav2Vec2-based speaker identification model on all 40 speakers. Retain set accuracy: 93.45%. Target speaker (forget set) accuracy: 99.37%. The model knows everyone very well. This is what we now need to selectively modify.
Stage 2
Gradient-Based Weight Zeroing
Instead of removing data, we identified which specific weights inside the model are responsible for recognizing the target speaker. We used gradient analysis to find the top 5% of weights most influential for that speaker's predictions — then zeroed them out. Like surgically cutting the exact wires that hold that person's memory. Then we ran recovery fine-tuning on the retain set. Result: forget accuracy dropped from 99% to 79%. Partial forgetting — promising but not complete yet.
Stage 3
Gradient Ascent Unlearning
A different strategy. Normal training reduces error. Gradient ascent does the opposite — we pushed the model's error upward on the target speaker's data. Essentially training the model to be wrong about that person. Forget accuracy dropped to 5.7% — very close to complete. But we had to be careful: push it too far and the model starts forgetting other speakers too. That is called catastrophic forgetting. We stopped at the right balance point.
Stage 4
Comparative Unlearning — 4 Methods
We implemented four completely different unlearning strategies and benchmarked them side by side.
Results — All Methods Compared
MethodForget Acc ↓Retain Acc ↑
Random Labeling9.76%99.15%
SISA Partitioning0%93.43%
Knowledge Distillation0%90.68%
⭐ Weight Editing0%98.31%
Gradient Ascent5.70%90.90%
💡 Winner: Weight Editing — complete forgetting at 0% with highest retain accuracy at 98.31%. Best of both worlds.
What Each Method Did
Random Labeling
Assign random labels from other speakers to the target speaker's data and fine-tune. Simple, but the model got confused rather than truly forgetting. Still had 9.76% forget accuracy — not good enough for GDPR.
SISA Partitioning
Split training data into 3 shards. Retrain only the shards containing the target speaker — but with that person removed. Achieved complete forgetting in just 14 seconds. Only method with a mathematical guarantee — data physically never seen in retrained shards.
Knowledge Distillation
Train a teacher model that has never seen the target speaker. Make the original model copy the teacher. Since the teacher never knew that person, the student inherits that ignorance. Target speaker confidence dropped from 99.2% to near 49% — essentially random chance.
Weight Editing — WINNER ⭐
Refined gradient zeroing. Zero the top 5% of influential weights, then do targeted recovery fine-tuning on the retain set. Result: 0% forget accuracy, 98.31% retain accuracy. Complete forgetting with almost no collateral damage.
Noise Stress Testing — Real-World Validation

We did not stop at clean audio. In the real world, voice systems deal with background noise, microphone distortion, and environmental sounds. We asked: if we add noise to the audio, will the forgotten speaker accidentally get recognized again?

We injected noise at multiple levels into both raw audio waveforms and the model's internal embeddings.

Key finding: Even under heavy noise, the forgotten speaker's accuracy stayed locked at 0%. The privacy boundary never broke under any noise condition tested.

ConditionBefore UnlearningAfter Unlearning

💡 The model didn't just forget — it forgot permanently and robustly under all conditions.
Key Conclusions
  • GDPR compliance is achievable — "Right to be forgotten" can be honored without full retraining
  • Weight Editing is optimal — 0% forget accuracy, 98.31% retain accuracy, best balance
  • Forgetting is permanent and robust — noise cannot reactivate the forgotten identity
Why Samsung funded this
Samsung builds voice-enabled products — phones, smart TVs, Bixby, Galaxy devices. Any product that learns user voice data must honor data deletion at scale. This research gives them a practical, efficient way to do that.
What was your specific role?
I was part of a three-member team under Samsung's PRISM program. My contributions covered implementing the unlearning pipeline stages, running the comparative evaluation across all four methods, conducting the noise robustness analysis, and contributing to the research documentation. We completed all milestones from February to May 2026 and are currently preparing an IEEE journal paper based on the findings.

⚛️ React.js — Core Concepts
What is React? Why use it?
React is a JavaScript library built by Facebook for building user interfaces. Instead of directly manipulating the browser's real DOM (which is slow and expensive), React maintains a Virtual DOM in memory. When state changes, React calculates the minimum changes needed and updates only those parts in the real DOM. It also promotes reusable components — small, self-contained UI blocks you compose together to build complex interfaces. This makes UIs fast, modular, and maintainable.
What is the Virtual DOM? How does reconciliation work?
The browser's real DOM is slow to update. React creates a lightweight copy of it in memory — the Virtual DOM. When state changes: (1) React updates the Virtual DOM, (2) compares the new Virtual DOM to the previous one — this is called diffing, (3) calculates the minimum number of real DOM operations needed, (4) applies only those targeted changes. Result: fewer, precise DOM updates = faster UI. This process is called reconciliation.
What are React Hooks? List the most important ones.

Hooks let you use React features (state, lifecycle, context) inside functional components — introduced in React 16.8 to replace class components.

useState — Manages local component state. const [count, setCount] = useState(0). Calling the setter triggers a re-render with the new value.

useEffect — Runs side effects after rendering — API calls, subscriptions, DOM manipulation. Dependency array controls when it runs: [] = only on mount, [value] = when value changes, no array = every render. Return a cleanup function to handle unmount.

useContext — Reads a React Context value directly without prop drilling (passing props through every layer). Great for auth state, theme, language.

useCallback — Memoizes a function reference so it isn't recreated on every render. Prevents unnecessary re-renders of child components that receive it as a prop.

useMemo — Memoizes a computed value. Only recalculates when dependencies change. Use for expensive calculations.

useRef — Holds a mutable value that persists across renders without triggering re-render. Also used to directly access DOM elements (e.g., focus an input).

Props vs State — what's the difference?

Props: Data passed from parent to child. Read-only — the child cannot modify them. Flow down the component tree. Like function arguments.

State: Data owned and managed by the component itself. Mutable via useState. When state changes, the component re-renders. Like a component's internal memory.

Rule: if data comes from outside → props. If data changes inside the component → state.

Controlled vs Uncontrolled component?

Controlled: Form input's value is driven by React state. Every keystroke updates state; the input's value prop is set from state. You have full control and can validate in real time. Most common approach.

Uncontrolled: Input manages its own value in the DOM. You use a useRef to read the value when needed (e.g., on submit). Less React code but less control — used for simple/legacy cases.

What is React Router?
A library for client-side routing in React apps. Instead of the browser fetching a new HTML page from the server on every link click, React Router intercepts navigation and renders different components for different URL paths — all without a full page reload. This is what makes React a Single Page Application (SPA): it behaves like a multi-page site but only loads once.
💡 Bonus: mention React.memo (prevents re-render if props haven't changed), lazy() + Suspense (code splitting for performance) if asked about optimization.
🟢 Node.js & Express.js — Core Concepts
What is Node.js?
Node.js is a runtime environment that lets you run JavaScript outside the browser — on a server. It is built on Google's V8 engine (same engine Chrome uses). Before Node.js, JavaScript only ran in browsers. Node.js made it possible to use JavaScript for backend development, enabling the full MERN stack in one language.
What is the Event Loop? (Most asked Node.js question)

Node.js is single-threaded — it runs on one thread. But it can handle thousands of concurrent requests without blocking. How? Through the event loop.

When Node receives a request, instead of waiting (blocking) for slow operations like database queries or file reads, it says "go do this, call me back when done" and moves on to the next request. When the operation completes, its callback is placed in a queue. The event loop continuously checks this queue and executes callbacks when the call stack is empty.

This is non-blocking I/O — Node never sits idle waiting for one operation to finish before starting another. This is why Node is highly efficient for I/O-heavy applications like REST APIs.

Order: Call Stack → Web APIs/libuv (async work happens here) → Callback Queue → Event Loop picks from queue → back to Call Stack.

Callbacks vs Promises vs Async/Await?

All three handle asynchronous operations — they evolved over time to solve the same problem more cleanly.

Callbacks: Pass a function to be called when async work is done. Problem: deeply nested callbacks become unreadable — "callback hell."

Promises: An object representing a future value — pending → fulfilled or rejected. Chainable with .then().catch(). Much cleaner than callbacks.

Async/Await: Syntactic sugar over Promises. Makes async code look synchronous — easier to read and debug. async marks a function as asynchronous; await pauses that function until the Promise resolves. Under the hood it is still Promises — no new runtime mechanism.

What is middleware in Express.js?
Middleware is a function that sits between the incoming request and outgoing response in Express. It has access to req, res, and next(). Middleware can: modify req/res, end the request-response cycle, or call next() to pass control forward. Examples: express.json() to parse JSON bodies, auth checking middleware, CORS headers, logging, error handling. Middleware runs in the order it is defined — arrangement matters.
What is JWT and how does authentication work with it?

JWT (JSON Web Token) is a compact, self-contained token for securely transmitting identity information. Structure: Header.Payload.Signature — three base64-encoded parts joined by dots.

Flow: User logs in → server verifies credentials → server signs a JWT with a secret key → sends token to client → client stores it (localStorage or httpOnly cookie) → client sends token in Authorization: Bearer <token> header with every request → server verifies the signature → grants or rejects access.

Key property: Stateless — the server stores no session data. The token itself carries user identity. This is what makes JWT ideal for scalable REST APIs.

💡 Key buzzwords to use: event-driven, non-blocking I/O, single-threaded, V8 engine, npm (Node Package Manager), middleware pipeline, stateless REST server.
🍃 MongoDB — Core Concepts
What is MongoDB? How is it different from SQL?

MongoDB is a NoSQL database that stores data as flexible JSON-like documents (internally BSON — Binary JSON). Instead of rows in tables, you have documents in collections.

FeatureSQL (MySQL, PostgreSQL)MongoDB (NoSQL)
Data formatRows & columns in tablesJSON-like documents
SchemaFixed, predefinedFlexible, schema-less
RelationsForeign keys + JOINsEmbedded docs or $lookup
ScalingVertical (bigger server)Horizontal (more servers)
Best forComplex queries, strict relationsFlexible data, high volume
What is Indexing in MongoDB?
Without an index, MongoDB must scan every document in a collection to find matches — a full collection scan, O(n). An index is like a book's index — it stores a sorted reference to field values with pointers to their documents. With an index, MongoDB jumps directly to matches — O(log n). You create one with db.collection.createIndex({field: 1}). Trade-off: indexes speed up reads but slightly slow down writes, because the index itself must be updated on every insert/update.
What is the Aggregation Pipeline?
MongoDB's way of processing and transforming documents through a series of stages. Each stage takes documents as input, transforms them, passes results to the next. Common stages: $match (filter — like SQL WHERE), $group (group by field — like SQL GROUP BY), $sort, $project (select/rename/reshape fields), $limit, $lookup (join with another collection). Used for analytics dashboards, reports, summaries — anything beyond simple find queries.
What is Mongoose?
Mongoose is an ODM (Object Data Modeling) library for MongoDB with Node.js. Even though MongoDB is schema-less, Mongoose lets you define a Schema in Node code to enforce structure, data types, and validation. You define a Schema → create a Model from it → use the Model to query, create, and update documents. It also provides middleware hooks (pre/post save), virtuals, and query helpers. Bridges MongoDB's flexibility with the structure Node.js applications need.
When would you use SQL over MongoDB?
SQL is better when: data is highly relational (many entities referencing each other), you need complex multi-table JOINs, you need full ACID compliance, or the schema is stable and well-defined. MongoDB is better when: data structure varies or evolves rapidly, you need horizontal scaling, you're storing flexible document-like records (user profiles, call logs, task data), or you want seamless JSON flow from frontend to backend — exactly why it fits the MERN stack.
💡 Connect to your projects: "I chose MongoDB because my data — call logs, task records, vulnerability reports — had varying structure and needed flexible document storage that integrates naturally with Node.js and REST APIs."
🔗 REST APIs — Core Concepts
What is a REST API?
REST (Representational State Transfer) is an architectural style for building web APIs. It defines how clients (browsers, mobile apps) communicate with servers over HTTP. You define endpoints — URLs that represent resources (like /users, /orders) — and use standard HTTP methods to perform operations on them. REST is not a library or protocol — it is a set of design constraints that make APIs predictable, scalable, and stateless.
Core REST Principles?

Stateless: Each request must contain all information needed to process it. The server stores zero client session data. This is what makes REST APIs horizontally scalable — any server can handle any request.

Uniform Interface: Consistent, predictable URL structure and HTTP method usage. /users = collection, /users/123 = specific user. Nouns in URLs, not verbs.

Client-Server separation: Frontend and backend are fully independent and communicate only through API calls. Each can evolve separately.

Cacheable: Responses can be cached by clients or CDNs for performance.

HTTP Methods — what each one means

GET — Retrieve data. No request body. Safe (no side effects) and idempotent (same result every call). GET /users returns all users.

POST — Create a new resource. Has a request body. Not idempotent — calling twice creates two records. POST /users creates a user.

PUT — Replace the entire resource. Send all fields. Idempotent. PUT /users/123 completely replaces that user.

PATCH — Partially update a resource. Only send the fields to change. More efficient than PUT for small edits.

DELETE — Delete a resource. DELETE /users/123 removes that user.

HTTP Status Codes — must-know

200 OK — Success. 201 Created — Resource created (POST). 204 No Content — Success, no body (DELETE).

400 Bad Request — Invalid client data, validation error. 401 Unauthorized — Not authenticated, missing or invalid token. 403 Forbidden — Authenticated but lacks permission. 404 Not Found — Resource doesn't exist.

500 Internal Server Error — Something crashed on the server. 503 Service Unavailable — Server overloaded or down.

REST vs GraphQL?

REST: Multiple endpoints, each returns a fixed data structure. Simple, widely understood. Can lead to over-fetching (more data than needed) or under-fetching (need multiple requests for related data).

GraphQL: Single endpoint (/graphql). Client specifies exactly what fields it needs in the query. Server returns only that — no more, no less. Great for complex nested data and mobile clients. Steeper learning curve.

Use REST: standard CRUD, simple relationships, most teams. Use GraphQL: complex nested queries, many different clients needing different data shapes.

🌿 Git & GitHub — Core Concepts
What is Git? What is GitHub? How are they different?

Git is a distributed version control system — a tool that tracks every change to your code over time, runs locally on your machine, and lets you revert to any version, create branches, and collaborate. The full history lives in your local .git folder.

GitHub is a cloud platform for hosting Git repositories. It adds collaboration features: pull requests, code reviews, issue tracking, Actions (CI/CD). GitHub is to Git what YouTube is to video — the platform, not the tool itself.

Essential Git Commands — explained plainly

git init — Initialize a new repo in the current folder. Creates a hidden .git directory.

git clone [url] — Copy a remote repository (full history included) to your local machine.

git add [file] — Stage changes for the next commit. Think of staging as "what I want in my next snapshot."

git commit -m "message" — Save a snapshot of staged changes. Commits are permanent checkpoints.

git push origin main — Upload local commits to the remote repo on GitHub.

git pull — Fetch remote changes and merge them into your current branch.

git branch [name] — Create a new branch for isolated development.

git checkout [branch] — Switch to a different branch.

git merge [branch] — Merge another branch into your current one.

git stash — Temporarily shelve uncommitted changes so you can switch branches cleanly. git stash pop restores them.

git log — View commit history. git status — See staged, modified, and untracked files.

What is a Pull Request (PR)?
A Pull Request is a GitHub feature where you request to merge your feature branch into the main branch — after teammates have reviewed your code. You push your branch, open a PR on GitHub describing what changed and why, reviewers comment and request changes, and once approved it merges. PRs are the standard professional collaboration workflow — they enforce code review and protect the main branch from direct unreviewed pushes.
What is a merge conflict and how do you resolve it?
A conflict occurs when two branches changed the same line of the same file and Git cannot automatically decide which version to keep. To resolve: open the conflicted file — Git marks both versions with <<<<<<, =======, >>>>>>. Edit the file to keep the correct version, delete the conflict markers, git add the file, then git commit. Best practice: communicate with teammates before working on the same file simultaneously.
What is .gitignore?
A .gitignore file tells Git which files and folders NOT to track. Common entries: node_modules/ (huge, regenerable with npm install), .env (environment variables and secrets — never commit secrets), build/ and dist/ folders (generated files), OS-specific files like .DS_Store.

🤖 AI / LLM — Core Concepts
What is an LLM (Large Language Model)?
An LLM is an AI model trained on enormous amounts of text — books, websites, code, research papers — to understand and generate human language. It learns statistical patterns of how words and concepts relate to each other. Given an input (called a prompt), it predicts and generates the most likely continuation or response. GPT-4, Claude, Llama, and Gemini are LLMs. The "large" refers to billions of learnable parameters (weights) in the neural network.
How does an LLM work — high level?

LLMs use the Transformer architecture (Google's 2017 "Attention Is All You Need" paper).

Tokenization: Text is split into tokens — roughly words or word pieces. The model processes tokens, not characters. "Hello world" = ~2 tokens; "unbelievable" = ~3 tokens.

Attention mechanism: The model learns which tokens are most relevant to each other. "The bank was robbed" — "bank" relates to "robbed", so it's a financial bank, not a river bank. Attention captures this contextual meaning across the entire input.

Prediction: Given input tokens, the model predicts the next most probable token, appends it, predicts the next — building the response one token at a time. This is called autoregressive generation.

Training: Trained on massive datasets by predicting the next token, comparing to the actual token, and adjusting billions of weights via backpropagation. Done billions of times.

What is Prompt Engineering?
Prompt engineering is crafting input text to get the best possible output from an LLM. The phrasing of your question significantly affects output quality. Techniques: clear instructions, providing examples (few-shot prompting), specifying output format ("respond only in JSON"), setting context with a system prompt, breaking complex tasks into steps (chain-of-thought). In your Sales Analytics Platform — you used structured system prompts to make the LLM extract call summaries in a consistent, parseable format.
System Prompt vs User Prompt?

System Prompt: Developer-defined instructions that set the AI's behavior, role, tone, and constraints — given before the conversation starts. Example: "You are a sales call analyzer. Extract the key decisions, follow-up actions, and sentiment from the transcript. Return as a JSON object with fields: summary, actions[], sentiment."

User Prompt: The actual runtime input — the call transcript text in your case. Good system prompts constrain output format and reduce unpredictability.

What is Temperature in LLMs?
Temperature controls the randomness of the model's output. Low (0.0–0.3): deterministic and focused — always picks the most probable token. Best for data extraction, summarization, structured output. High (0.7–1.0): creative and varied — explores less likely but interesting paths. Best for writing, brainstorming, ideation. In your Sales Analytics project, you'd use low temperature for consistent, structured call summaries.
What is OAuth 2.0? (Your AI Task Agent used this)
OAuth 2.0 is an authorization framework that lets a third-party app access a user's resources on another service — without the user sharing their password. Flow: User clicks "Sign in with Google" → your app redirects to Google's auth server → user grants specific permissions (scopes) → Google gives your app an access token → your app uses that token to call Gmail API / Calendar API on the user's behalf. The token has limited scope and expiry. Your AI Task Agent used this so users could safely authorize email and calendar access — you never stored their password.
What is WebRTC? (Your Sales Platform used this)
WebRTC (Web Real-Time Communication) is a browser API and protocol that enables peer-to-peer audio, video, and data sharing directly between browsers — without a central media server. Works over UDP for low latency. Requires a signaling server (your Node.js backend) to help two peers discover each other and exchange connection parameters (SDP offer/answer). Once connected, media flows directly peer-to-peer. You built this signaling layer in your Sales Call Platform to enable in-app calling.
What is RAG (Retrieval-Augmented Generation)?
RAG gives an LLM access to external, up-to-date information at query time. Flow: (1) search a vector database for documents relevant to the user's query, (2) inject those documents as context into the prompt along with the question, (3) the LLM generates an answer grounded in those documents. Solves the problem of LLMs having outdated training data or no access to private/specific documents. Used in enterprise AI systems, document Q&A, knowledge bases.

📦 Object-Oriented Programming (OOP)
What is OOP? What are its 4 pillars?
OOP is a paradigm that organizes code around objects — entities that combine data (attributes/fields) and behavior (methods/functions). The 4 pillars:
Pillar 1
Encapsulation — Hide internal details
Bundle data and the methods that operate on it inside a class. Control access with modifiers: private (only within the class), public (accessible from anywhere), protected (within class + subclasses). External code only interacts through defined public methods. Example: a BankAccount class exposes deposit() and withdraw() but keeps balance private — nobody can set balance = 1000000 directly. Prevents accidental or unauthorized modification.
Pillar 2
Abstraction — Show only what matters
Hide implementation complexity; expose only the necessary interface. When you call car.start(), you don't know or care about fuel injection, ignition timing, spark plugs — you just know it starts. In code: abstract classes and interfaces define what a class must do without specifying how. Reduces complexity for the caller and allows internals to change without affecting external code.
Pillar 3
Inheritance — Reuse and extend
A child class inherits properties and methods from a parent class, then adds or overrides as needed. Promotes code reuse. Example: Animal has eat() and sleep(). Dog extends Animal and adds bark(). Dog automatically inherits eat() and sleep() without rewriting them. Types: single, multilevel, hierarchical. Multiple inheritance is supported in Python but not Java (Java uses interfaces instead).
Pillar 4
Polymorphism — Same name, different behavior
Poly = many, morph = form. Same method name behaves differently based on the object or arguments. Two types: Method Overloading (compile-time) — same method name, different parameters in the same class: add(int, int) and add(double, double). Method Overriding (runtime) — child class redefines a parent's method with the same signature. When the overridden method is called on a child object, the child's version runs — even if the reference variable is of the parent type.
Abstract Class vs Interface?

Abstract Class: Can have abstract methods (no body) AND concrete methods (with implementation). A class can extend only one abstract class. Used when you want shared partial implementation. Has a constructor. Can have instance variables.

Interface: All methods are abstract (no implementation in Java pre-8; default methods allowed in Java 8+). A class can implement multiple interfaces — this is how Java achieves multiple inheritance. Used to define a contract: "any class implementing this must have these methods." No constructor, no instance variables (only constants).

Rule of thumb: abstract class = "is-a" relationship with shared code. Interface = "can-do" capability contract.

Method Overloading vs Method Overriding?

Overloading: Same class. Same method name, different parameters (number or type). Resolved at compile time. print(int x) and print(String s) — compiler picks based on argument type.

Overriding: Parent + child class. Child redefines parent's method with same name and same signature. Resolved at runtime based on the actual object type. Use @Override annotation in Java to explicitly confirm intent.

What is a Constructor?
A special method that runs automatically when a new object is created (new ClassName()). Initializes the object's attributes. Same name as the class. No return type. Can be overloaded. If you don't define one, the compiler provides a default no-argument constructor. The first line in a subclass constructor is often super() — calling the parent's constructor.

📊 Data Structures & Algorithms
What is Big O Notation?

Big O describes how an algorithm's time or space requirements grow as input size (n) grows. It measures worst-case efficiency — helps compare algorithms without caring about hardware.

O(1) — Constant: Same time regardless of n. Example: array[5] — index access is instant.

O(log n) — Logarithmic: Halves the problem each step. Binary search. Efficient even for huge n.

O(n) — Linear: Grows proportionally. Looping through an array once.

O(n log n): Merge sort, quick sort. The gold standard for sorting.

O(n²) — Quadratic: Nested loops over same data. Bubble sort. Gets slow fast — avoid for large inputs.

O(2ⁿ) — Exponential: Doubles with each addition to n. Recursion without memoization. Impractical for n > 30.

Key Data Structures — when to use each?

Array: Indexed collection. O(1) read by index, O(n) insert/delete in middle. Use when size is known and you need fast index access.

Linked List: Nodes with data + pointer to next. O(n) access, O(1) insert/delete at known position. Use when frequent insertions/deletions, size unknown.

Stack (LIFO): Push and pop from top. Last In First Out. Use for: function call stack, undo operations, parsing, DFS traversal.

Queue (FIFO): Enqueue/dequeue. First In First Out. Use for: BFS traversal, task scheduling, print queues.

Hash Map: Key-value pairs. O(1) average for insert/search/delete. The most versatile data structure in interview problems — use it when you need fast lookup by key, frequency counting, or caching.

Binary Search Tree (BST): Left child < root < right child. O(log n) search in a balanced tree. Use for: sorted data with fast search, range queries.

Common Problem-Solving Patterns

Two Pointers: Two indices moving toward each other or in the same direction through an array. Reduces O(n²) brute force to O(n). Classic use: find pair with target sum in a sorted array, reverse an array in-place.

Sliding Window: Maintain a variable-size window over a subarray/substring. Expand right, shrink left as needed — O(n). Classic use: maximum sum subarray of size k, longest substring without repeating characters.

Binary Search: On any sorted structure — compare midpoint, eliminate half each time — O(log n). Think binary search whenever the input is sorted or you're finding a threshold/boundary value.

HashMap for O(1) lookup: When brute force requires nested loops (O(n²)), ask: can storing seen values in a hashmap reduce it to O(n)? Classic: Two Sum — instead of checking every pair, store each number and check if the complement exists in the map.

BFS vs DFS: BFS (queue) — explore level by level, optimal for shortest path. DFS (stack/recursion) — go deep first, good for exploring all paths, connected components, cycle detection.

Sorting Algorithms — key ones

Bubble Sort: Repeatedly swap adjacent elements if out of order. O(n²) time, O(1) space. Simple to code, terrible for large data. Never used in production.

Insertion Sort: Build sorted array one element at a time. O(n²) worst, O(n) best (already sorted). Good for small or nearly-sorted arrays.

Merge Sort: Divide in half, sort each half, merge. O(n log n) always. O(n) space. Stable (preserves order of equal elements). Best for large datasets.

Quick Sort: Pick a pivot, partition into < pivot and > pivot, recurse. O(n log n) average, O(n²) worst. O(log n) space. Very cache-friendly — fastest in practice for most inputs. Most language built-ins use a variant.

💻 CS Fundamentals — OS, DBMS, CN
Process vs Thread?

Process: An independent program in execution. Has its own memory space (code, data, heap, stack). Isolated — one process crashing doesn't affect others. Heavier to create and context-switch.

Thread: A unit of execution within a process. Shares memory with other threads in the same process. Lighter — faster to create and switch. Multiple threads can run in parallel on multi-core CPUs. Risk: shared memory → race conditions if not synchronized with locks/mutexes.

Node.js note: single main thread, but I/O is handled by underlying worker threads and the OS — giving the appearance of concurrency through the event loop without race conditions in your application code.

What is a Deadlock?
Deadlock is when two or more processes are each waiting for a resource held by the other — and none can proceed. Classic example: Process A holds Resource 1 and waits for Resource 2. Process B holds Resource 2 and waits for Resource 1. Both wait forever. Prevention strategies: resource ordering (always acquire locks in a fixed global order), timeout mechanisms, deadlock detection and recovery algorithms.
What is ACID in databases? (Very commonly asked)

ACID is a set of guarantees that make database transactions reliable.

Atomicity: A transaction is all-or-nothing. If any part fails, the entire transaction rolls back — no partial updates. "Transfer $100 from A to B" — either both the debit and credit happen, or neither.

Consistency: A transaction moves the database from one valid state to another valid state. All defined rules and constraints are maintained.

Isolation: Concurrent transactions execute as if they were sequential. One transaction's intermediate state is invisible to others running simultaneously.

Durability: Once committed, the transaction persists — even if the system crashes immediately after. Written to disk, not just memory.

SQL databases are fully ACID. MongoDB is ACID within a single document; multi-document transactions are supported since v4.0 but add overhead.

What is Database Normalization?
Normalization organizes a relational database to reduce data redundancy (storing same data multiple times) and improve integrity (one source of truth). It divides data into related tables with defined relationships. 1NF: atomic values in each column, no repeating groups. 2NF: 1NF + no partial dependencies on composite keys. 3NF: 2NF + no transitive dependencies (non-key columns don't depend on other non-key columns). Goal: each fact stored exactly once in the right place.
SQL Joins — types?

INNER JOIN: Returns rows that have matching values in both tables. Most common — only matching rows.

LEFT JOIN: All rows from the left table + matched rows from the right. Unmatched right rows show NULL.

RIGHT JOIN: All rows from the right table + matched left rows.

FULL OUTER JOIN: All rows from both tables. NULLs where no match exists.

HTTP vs HTTPS?
HTTP — HyperText Transfer Protocol. Data travels as plain text. Anyone intercepting the network traffic can read it. HTTPS — HTTP with SSL/TLS encryption. Data is encrypted end-to-end. The server has an SSL certificate signed by a Certificate Authority (CA). The browser verifies this before establishing a secure connection. Always use HTTPS in production — browsers now actively warn users about HTTP sites, and modern APIs reject non-HTTPS requests.
TCP vs UDP?

TCP (Transmission Control Protocol): Connection-oriented. Establishes connection via 3-way handshake. Guarantees delivery, ordering, and error checking. Retransmits lost packets. Slower due to overhead. Use for: HTTP/S, file transfer, email, database connections — anywhere data integrity matters.

UDP (User Datagram Protocol): Connectionless. No delivery guarantee, no ordering. Faster, minimal overhead. Use for: video streaming, online gaming, live voice/video calls, DNS lookups — where speed and low latency matter more than perfect delivery. WebRTC (which you used) uses UDP internally for low-latency audio/video.

💡 Process vs Thread and ACID are the most commonly asked CS fundamentals in interviews. Know them cold. If asked about OS scheduling or memory management — answer what you know, be honest about depth.
